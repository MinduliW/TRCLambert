# Tiny Recursive Models for solving the **$J_2$-perturbed Lambert problem**.

This repo accompanies the paper *Tiny Recursive Models for Solving the $J_2$-Perturbed
Lambert Problem* (Wijayatunga & Armellin). It implements **TRM-PL**, a compact,
weight-shared neural solver that takes a Keplerian Lambert departure velocity and
refines it — through repeated $J_2$ propagation and error feedback — into the velocity
that actually reaches the target under perturbed dynamics. The same small reasoning
module is applied for $K$ iterations, so accuracy comes from iteration depth rather than
parameter count (~2.3–2.6 M params).

The solver is trained end-to-end with a differentiable RK4 propagator in the loop, and
is compared against a classical Newton-shooting baseline and the DNN-plus-shooting
pipeline of Yang et al. (2022).

## Installation

Python 3.9+ with PyTorch. The core dependencies are:

```bash
pip install torch numpy scipy matplotlib h5py lamberthub
```

- `torch` — model and differentiable RK4 propagator
- `lamberthub` — classical Keplerian Lambert solver (initial guess)
- `h5py` / `scipy` — loading MATLAB `.mat` datasets (e.g. the Jovian set from Yang et al.)
- `numpy`, `matplotlib` — data handling and figures

A GPU is recommended; the LEO cases train in roughly 6–8 h and the Jovian case in
12–16 h on a single RTX 3080, dominated by RK4 propagation over the long Jovian arcs.

## 1. Generate datasets

Datasets are generated by forward sampling: orbital elements are drawn, propagated under
$J_2$ to get $(\mathbf{r}_1, \mathbf{r}_2, \Delta t)$, then the Keplerian Lambert problem
is solved to get the (biased) initial guess. Each sample stores both the Lambert velocity
and the true $J_2$-consistent velocity.

```bash
cd DataGeneration
python generate_datasets.py
```

This writes a train and validation `.npz` for each case (`leo_single`, `leo_multi`,
`jovian`). Sample counts and seeds are set by the constants at the top of
`generate_datasets.py` (`N_TRAIN`, `N_VAL`, `SEED_TRAIN`, `SEED_VAL`).

The trainer reads its data from `data/` at the repo root, expecting:

| Dataset arg  | Train file                              | Val file                              |
|--------------|-----------------------------------------|---------------------------------------|
| `single_rev` | `data/leo_single_train_lambertpy.npz`   | `data/leo_single_val_lambertpy.npz`   |
| `multi_rev`  | `data/leo_multi_train_lambertpy.npz`    | `data/leo_multi_val_lambertpy.npz`    |
| `jovian`     | `data/jovian_train_lambertpy.npz`       | `data/jovian_val_lambertpy.npz`       |

Place (or symlink) the generated `*_lambertpy.npz` files under `data/` at the root so the
paths above resolve.

## 2. Train

```bash
python lambert_trc_j2.py <variant> <dataset> [options]
```

`<variant>` and `<dataset>` are positional and optional (defaults: `pos_only single_rev`).

**Variants** map to the paper as follows:

| CLI variant       | Paper        | Description                                                        |
|-------------------|--------------|--------------------------------------------------------------------|
| `learned_lambert` | Variant A    | Head 1 learns the Lambert solution; Head 2 refines it             |
| `vel_supervised`  | Variant B    | Refine from the classical Lambert solution; supervise position **and** corrected velocity |
| `pos_only`        | Variant C    | Refine from the classical Lambert solution; supervise position only (no velocity labels) |

(`pos_only_residual` and `vel_supervised_residual` are residual-head variants for ablation.)

**Datasets:** `single_rev`, `multi_rev`, `jovian`.

Common options:

| Flag                 | Default        | Meaning                                                        |
|----------------------|----------------|----------------------------------------------------------------|
| `--epochs`           | `300`          | Stage-2 training epochs                                        |
| `--K`                | per-dataset    | Outer refinement iterations (3 for `single_rev`, 4 otherwise)  |
| `--lr`               | `1e-4`         | Learning rate                                                  |
| `--resume`           | off            | Resume from `checkpoints/<run_name>_last.pt`                   |
| `--resume-from PATH` | —              | Resume from a specific checkpoint                              |
| `--skip-stage1`      | off            | Skip Head-1 pretraining (Variant A)                            |
| `--joint`            | off            | Add an $L_{v_0}$ term supervising Head 1 jointly with Head 2   |
| `--max-step`         | per-dataset    | RK4 max step (s) for the propagator                            |
| `--train-max-step`   | auto           | Coarser RK4 step during training (speeds up Jovian)            |

Examples:

```bash
# Variant C, single-revolution LEO (the paper's most reliable solver)
python lambert_trc_j2.py pos_only single_rev

# Variant B, multi-revolution LEO
python lambert_trc_j2.py vel_supervised multi_rev --K 4

# Variant C, multi-revolution Jovian
python lambert_trc_j2.py pos_only jovian
```

Checkpoints are written to `checkpoints/`.

## 3. Evaluate and plot

```bash
# Head-1 (initial guess) evaluation
python LambertPy/eval_stage1.py

# Head-2 refinement result plots
python LambertPy/plot_stage2_j2.py

# Dataset statistics and distribution figures
python DataGeneration/analyze_dataset.py
python DataGeneration/plot_dataset.py
```

Reported metrics follow the paper: terminal-position error
$\lVert \mathbf{r}_2 - \mathbf{r}_f^{(K)} \rVert$ and corrected initial-velocity error
$\lVert \mathbf{v}_{1,\mathrm{true}} - \mathbf{v}_1^{(K)} \rVert$, with per-component
statistics and quantiles. Applying a single Newton corrector step on top of the TRM-PL
output further tightens the Jovian median to sub-100 m.

## Notes

- Trained weights and datasets (`*.pt`, `*.npz`, `*.mat`, `*.npy`) are **git-ignored** —
  they are large and regenerative. Regenerate datasets with `DataGeneration/` and retrain
  to reproduce checkpoints.
- The training propagator can use a coarser RK4 step than evaluation (`--train-max-step`);
  validation metrics always use the finer `--max-step` for fidelity.
- Code under `LambertPy/depreciated/` is kept for reference and is not part of the current
  pipeline.
